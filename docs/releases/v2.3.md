# 2026-06-27

Scope library release. v2.2 turned Daily Scholar into a multi-tenant app; **v2.3 turns "scope" from a single hidden per-user setting into a first-class, shareable, forkable entity**. New users land on an onboarding picker offering five system-owned starter scopes (ML, Physics, Biology & Life Sciences, Economics & Finance, Climate & Earth Sciences). Each user gets a library of saved scopes — owned + granted — and switches which one is active. Public scopes are searchable and forkable; private scopes are shareable via a request/approval workflow that mirrors the existing invite-code pattern. **One new migration (`0012_scopes`), three new tables (`scopes`, `scope_access_grants`, `scope_access_requests`), one new column (`user_settings.active_scope_id`), twelve new starter topic YAMLs (`config/topics/starter/`), zero new dependencies, zero env-var changes.** Solo mode (`__local__`) and the legacy `/user/scope` endpoint preserved end-to-end — every paper-discovery / quiz read keeps working unchanged.

## Highlights

### Scope as a first-class entity

A scope is now a row in the `scopes` table with `id`, `name`, `description`, `owner_user_id`, `visibility`, `scope_mode`, `scope_topic_ids`, `forked_from_scope_id`, and timestamps. Users can save many, switch between them, and share them — the same way topics work after the Phase C ownership refactor in v2.2.

`UserSettings.active_scope_id` (nullable FK to `scopes.id`) points at the row currently driving discovery / review / quizzes. `null` means "no scope picked yet" — the new `ScopePickerGuard` bounces the user to the onboarding picker until they pick one.

The legacy `UserSettings.scope_mode` / `scope_topic_ids` columns stay populated as a back-compat cache; the `set_active` service refreshes them whenever the user switches scopes so existing paper-discovery and quiz code that reads `GET /user/scope` keeps working without modification.

### Five system-owned starter scopes

Shipped under `config/topics/starter/`:

- **ML Starter** → existing `ml-foundations` + `generic-ml`
- **Physics Starter** → classical mechanics, quantum mechanics, statistical mechanics (3 new topics)
- **Biology & Life Sciences Starter** → molecular biology, cell biology, genetics & genomics (3 new topics)
- **Economics & Finance Starter** → microeconomics, macroeconomics, financial economics (3 new topics)
- **Climate & Earth Sciences Starter** → climate science, atmospheric physics, oceanography (3 new topics)

Twelve new `foundations`-style YAML topics with focused keyword lists (~30–40 each), domain-correct arXiv categories (so paper discovery actually filters to the right corner of arXiv), and structured `key_concepts` + `learning_objectives` blocks. New `backend/services/starter_scopes.py` runs at boot after `bootstrap_topics_from_yaml` and inserts any missing starter, refreshes `topic_ids` if the catalog has grown, and otherwise leaves user-editable description / mode alone. Idempotent. Also runnable manually via `scripts/seed_starter_scopes.py`.

### Public scopes are searchable and forkable

`GET /scopes/search?q=` runs an ilike substring match across `name + description` of every public scope (system starters + user-owned public scopes). Forking via `POST /scopes/{id}/fork` copies the source's `scope_mode` + `scope_topic_ids` into a new private row owned by the caller, with `forked_from_scope_id` stamped at the source for lineage. The FK uses `ondelete=SET NULL` so deleting a source scope doesn't cascade-kill its forks — lineage breaks gracefully, child rows survive.

Forks start private regardless of source visibility — the user can flip to public from the editor if they want. There's also a **"Use directly"** action on `/scopes/browse` that sets a public scope as active without forking, so the owner's edits propagate live (good UX for starter scopes a user just wants to consume).

### Private scopes are shareable by request + approval

`POST /scopes/{id}/access-requests` from a non-owner inserts a `pending` `ScopeAccessRequest` row. The owner sees it in `/scopes/access-requests/incoming` and decides via `POST /scopes/access-requests/{id}/decide` (body: `{"decision": "approve" | "deny"}`). Approving inserts a `ScopeAccessGrant`, which is what `can_view_scope` checks alongside ownership + public visibility. Denying just stamps `decided_at`; the requester can resubmit later (one *pending* per (scope, requester) — denied history is preserved). Mirrors the invite-code + admin-approval pattern from v2.2.

### Onboarding picker for new users

`/scopes/picker` greets new users with a 2-column grid of starter scopes, plus two equal-weight secondary paths: "Browse all public scopes" (links to `/scopes/browse`) and "Start from scratch" (creates an empty `mode=all` private scope, sets it active, routes to the editor). `<ScopePickerGuard />` in the layout bounces logged-in onboarded users with no active scope here automatically — composes cleanly with the existing `<AuthBoundary>` → `<OnboardingGuard>` chain. Skip prefixes (`/scopes/*`, `/settings/scope`, `/onboarding`, `/login`, `/signup`, `/account/*`) let users navigate inside the scope-setup surface without being yanked back.

### Frontend surface

- **`/settings/scope`** — rebuilt as the **library view**: active-scope banner, "My scopes" section (with Set Active / Delete / link-to-edit), "Shared with me" section (read-only relation), Create button (creates a draft + routes to editor).
- **`/settings/scope/[id]`** — new **per-scope editor**: name, description, mode picker, topic picker, visibility toggle, set active, fork (for non-owners), delete. Non-owners get a read-only view with a prominent Fork button.
- **`/scopes/browse`** — public-scope discovery: debounced search, per-result Fork or Use Directly, plus a "Request access by ID" form for private scopes shared out-of-band.
- **`/scopes/requests`** — two-section inbox (Incoming with Approve/Deny on pending, Outgoing for history), each with a status filter (`pending` / `approved` / `denied` / `all`).
- **`/scopes/picker`** — first-run picker (covered above).
- **`ScopeTour` bumped to v2** — targets `[data-tour="scope-active"]` + `[data-tour="scope-library"]` on the library page; the per-scope editor preserves the old `data-tour="scope-mode"` / `scope-save"` anchors for a future per-page tour.

### Migration of existing per-user scopes

`scripts/migrate_to_scope_library.py` walks every `UserSettings` row and, for each user with a real `users` row and no `active_scope_id` yet, creates a private `"My scope"` Scope owned by them carrying their existing `scope_mode` + `scope_topic_ids`, then points `active_scope_id` at it. Dry-run by default; pass `--apply` to commit. Users without a matching `users` row (the `__local__` sentinel and any other pre-auth handles) are skipped — they land on the onboarding picker at next sign-in. Idempotent.

In Docker, run inside the backend container so it talks to Postgres:

```bash
docker compose up --build -d
docker compose exec backend python scripts/migrate_to_scope_library.py            # dry-run
docker compose exec backend python scripts/migrate_to_scope_library.py --apply    # commit
```

## Schema changes

Migration `0012_scopes`:

- **`scopes`** — `id`, `name`, `description`, `owner_user_id` (FK users.id, nullable → NULL = system scope), `visibility`, `scope_mode`, `scope_topic_ids` (JSON), `forked_from_scope_id` (FK self, SET NULL on delete), timestamps. Indexes: `(owner_user_id, visibility)`, `(visibility, name)`.
- **`scope_access_grants`** — `(scope_id FK CASCADE, user_id String(100), granted_by_user_id FK users.id, granted_at)` with `UNIQUE(scope_id, user_id)`. One row per (scope, grantee).
- **`scope_access_requests`** — `(scope_id FK CASCADE, requester_user_id String(100), message, status, created_at, decided_at, decided_by_user_id)`. Multiple denied rows allowed per (scope, requester) for history; one-pending-per-pair enforced in the service layer.
- **`user_settings.active_scope_id`** — nullable FK to `scopes.id`, SET NULL on delete.

`scope_access_grants.user_id` and `scope_access_requests.requester_user_id` are kept as `String(100)` for consistency with the existing user-scoped tables (`topic_subscriptions`, `seen_papers`, …). Switching all of those to integer FKs is filed as post-beta tech debt — see the TODO at the top of the SCOPES section in `backend/database.py`.

`delete_scope` cleans up explicitly (clears `UserSettings.active_scope_id` pointers, breaks fork lineage, drops grants + requests, then deletes the row) rather than relying on the FK clauses, which SQLite doesn't enforce without `PRAGMA foreign_keys = ON`. Same convention as `topic_subscriptions.cleanup_subscriptions_for_topic`.

## Notes

- **No new env vars, no new Python deps, no new npm deps.** Everything builds on what v2.2 already shipped.
- **Solo mode preserved.** `__local__` is treated as admin for view + edit checks (same as the topic-ownership module's `resolve_caller`). The starter scopes are visible to it; the migration script skips it.
- **Search semantics.** Substring ilike on `name + description` — chosen to keep the first release small. Filtering by topic membership is filed in `FUTURE_FEATURES.md`.
- **Tests.** 41 new tests in `backend/tests/test_scopes.py` covering create / view / edit permissions, fork lineage, public search, library shape, the full access-request lifecycle, set/get/clear active, delete cascade, starter seeding, and migration idempotence.
