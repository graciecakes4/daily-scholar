# 2026-06-25

Mobile-navigation release. v2.0 hardened the prod deploy; v2.1 makes the app actually usable on a phone. The existing six-item horizontal top nav overflowed the iPhone viewport — Quiz and Settings sat off-screen and required horizontal scroll to reach. Worse, the dashboard rows themselves (stats band, section tabs, paper / topic-review / quiz action bars) were laying out wider than the viewport, so any accidental horizontal swipe could land on the screen and pan the page. v2.1 replaces the top nav with a **bottom tab bar on mobile only** (Home / Papers / Topics / Quiz / More) keyed to `usePathname()`, makes every overflowing dashboard row **responsive**, and adds an **`overflow-x: hidden` safety net** at `html, body` so future stray-width children can't reintroduce the sideways pan. Desktop layout (≥ `md`) is byte-equivalent to v2.0. One PR (#35) lands; 5 files changed, +262 / −45. No migrations.

## Highlights

**Bottom tab bar on mobile, top nav on desktop**
- New `frontend/components/MobileTabBar.tsx` client component renders a five-tab bar fixed to the bottom of the viewport at `md:hidden` (Home / Papers / Topics / Quiz / More). Active tab is derived from `usePathname()` so route changes light up the correct icon — Home matches `/`, the others match their `/papers`, `/topics`, `/quiz` prefixes, and any route not claimed by a tab (e.g., `/settings/*`) shows "More" as active. Tapping More opens a bottom sheet (`role="dialog" aria-modal="true"`) containing Settings + API Docs links; the sheet closes on route change, on Esc, and on scrim tap.
- `frontend/app/layout.tsx` — the existing horizontal nav links row is now `hidden md:flex`, so the top bar collapses to just the logo on phones. `<MobileTabBar />` is mounted globally and self-hides at `md:hidden`. `main` gets `pb-24 md:pb-8` so dashboard content clears the fixed bar, and the textual footer is `hidden md:block` because it would otherwise be obscured.
- Bottom bar padding honors `env(safe-area-inset-bottom)` for iOS home-indicator clearance on PWAs installed to the home screen. The "More" sheet uses `calc(env(safe-area-inset-bottom) + 1.25rem)` for matching clearance when expanded.

**Responsive dashboard rows (the actual overflow culprits)**
- **Stats band** — inner row was `flex items-center gap-6` over four stat blocks inside a `flex-wrap` parent; the inner row couldn't wrap and ran off-screen. Now `grid grid-cols-2 gap-3 md:flex md:items-center md:justify-between md:flex-wrap md:gap-4`, with "Best: N days" spanning both columns on mobile (`col-span-2 md:col-auto`). Each stat block gets `min-w-0 truncate` so a long localized label can't push the band wide again.
- **Section tabs (Today's Paper / Topic Review / Quiz)** — three buttons with icons + full labels + dot badges were a `flex gap-2` row that never shrank. Now `grid grid-cols-3 gap-1 md:flex md:gap-2` with abbreviated labels on mobile (`<span class="hidden md:inline">Today's </span>Paper`, same for "Topic Review" → "Review"). Dot / count badge gets `flex-shrink-0` so it never pushes the label off the button.
- **Paper action bar** (Open + PDF on the left; New paper + Save to Archive on the right) was `justify-between` and overflowed on mobile. Now `flex flex-col gap-2 md:flex-row md:items-center md:justify-between`, with the right-side group `flex-wrap` so a long localization can wrap to two lines instead of overflowing.
- **Topic review header** (course badge + topic name on the left; New / Save actions on the right) — same `justify-between` overflow pattern. Now `flex flex-col gap-3 md:flex-row md:items-start md:justify-between`, with `break-words` on the topic title and `flex-wrap` on the action group.
- **Quiz header** (Knowledge Check metadata + Save Results / New Quiz buttons) — identical fix: stack on mobile, flex row on `md+`, action group `flex-wrap`.

**Horizontal-scroll safety net**
- `frontend/app/globals.css` gains `html, body { overflow-x: hidden; max-width: 100vw }`. Any future stray-width child is clipped instead of producing a sideways scrollbar. Trade-off: visible content that genuinely exceeds the viewport gets clipped rather than scroll-revealed; in practice this is the preferred behavior on a mobile dashboard.
- Secondary motivation: with a fixed bottom tab bar, an accidental horizontal swipe near the bottom of the screen registers as a tap on the underlying tab — Grace was seeing the "More" sheet open mid-scroll because each pan landed on the fifth column. Killing horizontal pan at the page level removes the failure mode entirely.

**Cowork workspace folder excluded from git**
- `.gitignore` gains an entry for `daily scholar/` — the local cowork session workspace folder that sits inside the repo. Design explorations (the four-option mobile-nav mockup HTML from this PR, future scratch outputs) live there and stay on disk for reference but never enter the repo. Mirrors the existing pattern for `pr-scripts/`, `PUBLIC_REPO_AUDIT.md`, and `PWA_MIGRATION_PLAN.md`.

## Coming next

Followups noted during the mobile-nav pass:

- **iPhone hardware verification.** Mobile changes were verified in a desktop browser at narrow widths and pass `npx tsc --noEmit` clean. Worth a re-check on a physical iPhone — specifically: home-indicator clearance under the bottom bar, More-sheet drag-handle ergonomics, and scroll behavior with the fixed bar through a multi-page session — before promoting v2.1 to the broader beta cohort.
- **Other pages not audited.** Only the dashboard (`app/page.tsx`) had per-row overflow fixes applied. `/papers`, `/topics`, `/quiz`, and `/settings/scope` may still have their own overflow patterns under narrow viewports. The global `overflow-x: hidden` in `globals.css` keeps them usable in the meantime — anything that would have overflowed is now clipped — but the underlying rows should get the same responsive treatment as the dashboard.
- **Active-tab badges.** The bottom tab bar doesn't yet show notification dots / counts (the way the in-page section tabs do for the topic-review and quiz counts). Worth a follow-up once `getDailyContent` is callable at the layout level — would help drive the user back to Topics / Quiz when there's new daily content waiting.
- **Settings depth.** "More → Settings" currently lands on `/settings/scope`, which is the only settings page that exists. As more settings pages land (notification preferences, topic CRUD entry, push debug, account / CF Access linkage), the "More" sheet should grow into a proper grouped menu instead of two flat rows. Don't over-engineer it yet; revisit when there are 4+ entries.
