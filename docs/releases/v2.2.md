# 2026-06-26

Multi-user release. v2.0 hardened the prod deploy; v2.1 made it usable on a phone; **v2.2 turns Daily Scholar from a solo praxis tool into a real multi-tenant app**. Six feature PRs land in one release: configurable push notifications, the full multi-user auth foundation (in-app email+password signup, invite-gated admin approval, per-user topic ownership, topic discovery + subscriptions, an LLM-driven onboarding wizard, admin user-management UI), an append-only admin audit log, self-service password / username change + admin password reset, beta-launch hardening (rate limiting + CSRF + password-strength hints), and a per-page guided product tour with versioned server-side state. Plus a code-review agent playbook (`AGENTS.md`). **Eight new migrations (0004 → 0011), two new Python deps (`passlib`, `bcrypt`), one new npm dep (`driver.js`), three new env knobs.** Solo mode (`__local__`) preserved end-to-end — every change is additive to the existing local-dev experience.

## Highlights

### Configurable scheduled push notifications (PR #37)

Registry-driven notification types — study reminder, paper drop, weekly recap, quiz/review nudge — scheduled per-user via cron in a new `user_settings.notification_settings` JSON column. Adding a new notification type later is one entry in `notifications.REGISTRY`; the UI auto-renders a card for it (driven by `GET /notifications/types`). APScheduler jobs hold stable per-user IDs (`notif:<user>:<type>`) so toggling a type on/off takes effect immediately — no restart. `POST /notifications/test/{type}` button on `/settings/notifications` exercises the same dispatch path as the scheduler, so a green test means the cron will work too. Frontend page provides daily/weekly toggle + time picker + day-of-week + Preview + Test per card. Migration `0004_notification_settings` adds the JSON column.

### Multi-user auth foundation, Phases A–F (PR #38)

Six-phase rollout shipped in one PR — Daily Scholar gains real accounts on top of the existing CF Access path.

- **Phase A — In-app email+password auth.** New `users` + `sessions` tables (migration 0005), `POST /auth/signup`, `POST /auth/login`, `POST /auth/logout`, `GET /auth/me`. Server-side opaque session cookies (30-day TTL, HttpOnly, SameSite=Lax, Secure derived from `debug`). Refactored `get_current_user_id` to prefer the session cookie, fall back to CF Access header, fall back to `X-User-Id`, fall back to `__local__` sentinel — every existing solo / beta flow preserved. `scripts/create_admin.py` seeds the bootstrap admin. **`users.email` is the login credential; `users.user_id` is the foreign-keyable identity string** used in the existing 9 user-scoped tables — two-column identity split lets users pick a custom handle without refactoring every `user_id VARCHAR(100)` column. Frontend pages: `/login`, `/signup`, `/account/pending`, `/account/suspended`. New `<UserMenu />` in the nav; `<AuthBoundary />` redirects to `/login?next=...` on any 401.
- **Phase B — Invite codes + admin approval queue.** Migration `0006_invite_codes`. `/auth/signup` now requires an `invite_code` body field unless `OPEN_SIGNUP=1` (env-flagged off for dev / tests). New signups land in `pending` status; admin approves via `/admin/approvals` (flips to `active`). **New `require_admin` dependency replaces `require_cloudflare_access` on `/admin/*`** — closes the deferred admin role gate from v1.1. Solo `__local__` keeps passing through; real users need a `role='admin'` User row. First two tabs on `/settings/admin`: Pending Approvals (approve / reject) + Invite Codes (generate with optional expiry / max_uses, list, revoke, copy to clipboard).
- **Phase C — Per-user topic ownership + visibility.** Migration `0007_topic_ownership` adds `topics.owner_user_id` (nullable; `NULL` = system topic, e.g. Grace's praxis yaml-bootstrapped topics) and `topics.visibility` (`private` / `public`, defaults backfill `public` for existing rows). User-created topics get an opaque `usr-xxxxxx` id so two users can both name their topic "ML Papers" without slug collisions; yaml topics keep their human slugs. Topic CRUD enforces ownership (404 not 403 when the caller can't view — private topics don't leak via existence checks). `/topics/import-yaml` and `/topics/export-yaml` switched to admin-only.
- **Phase D — Topic discovery + subscriptions.** Migration `0008_topic_subscriptions`. New `/topics/discover` page + search endpoint over other users' public topics. `POST /topics/{id}/subscribe`, `DELETE /topics/{id}/subscribe`. **Scope rule changed: from "system OR own OR any public" to "system OR own OR (subscribed AND still public)"**. Owner flipping public→private silently drops the topic from subscribers' scope without deleting the subscription row — flips back to public, topic reappears. Live updates "for free" via FK reference. Topic catalog gains "Subscribed" emerald badge + Unsubscribe action on subscribed-from-others rows.
- **Phase E — LLM-driven onboarding wizard.** Migration `0009_user_onboarded` adds the flag (backfills existing rows to `true` so admins aren't bounced through the wizard). New `/onboarding` three-step page: describe interests → review LLM-drafted topic config (keywords + arXiv categories + key concepts) → save creates owned topic with `owner_user_id=current_user, visibility='private'` AND flips `onboarded=true` atomically. Skippable on every step. `<OnboardingGuard />` in the layout redirects logged-in unonboarded users. Defensive normalization on the LLM output (clamps list lengths, lowercases keywords, drops non-string items, three fallback paths to a token scaffold if the model errors).
- **Phase F — Admin account-management UI.** New `/admin/accounts` endpoints (list with status/role filters, `PUT /role` with last-admin protection, `PUT /status` with self-suspend refusal + session revocation on suspend). Third tab "Users" on `/settings/admin`: status/role badges, filter chips, Promote/Demote + Suspend/Reactivate per row with confirm dialogs.

Spans 7 commits, 8 alembic migrations (0005 + 0006 + 0007 + 0008 + 0009 within this PR), 100+ tests across the auth domain.

### Append-only admin audit log (PR #39)

Migration `0010_admin_audit_log` adds the `admin_audit_log` table with denormalized actor + target identifiers (so display info survives if the underlying user / invite row is deleted). `services/audit_log.log_event()` is best-effort — wrapped in try/except + warning log so a DB hiccup in the logger never blocks an admin action. Wired into all six admin mutations: approve / reject (with target email), role_change (with old→new metadata), suspend / reactivate (with status delta), invite create (with max_uses + expiry), invite revoke (only when actually flipped — no audit noise on double-revoke). `GET /admin/audit` with filters (event_type / actor / target_id / since / until) + pagination. New "Audit log" tab (4th) on `/settings/admin` with color-coded event badges + click-to-expand metadata JSON. **Worth knowing — naming gotcha:** SQLAlchemy's declarative `Base` reserves `metadata` as an instance attribute, so the model uses `audit_metadata = Column("metadata", JSON, ...)` — Python attr is `audit_metadata`, SQL column + JSON key stay `metadata`.

### Self-service password + username change + admin password reset (PR #40)

Three account-management surfaces on top of the auth foundation. **`PUT /auth/password`** — self-service password change, requires current password, revokes every OTHER session for the user (kicks hijacked devices) but preserves the current session via cookie token. **`PUT /auth/username`** — self-service handle change, requires current password, cascades the new `user_id` across all 10 string-`user_id` tables in one transaction (sessions FK on `users.id` int, so the current cookie keeps working post-rename; audit log identifiers freeze at-the-time). **`PUT /admin/accounts/{id}/password`** — admin reset, skips current-password check, revokes ALL target sessions, logs new `EventType.USER_PASSWORD_RESET_ADMIN` event with `{new_password_length: N}` in metadata (never the password itself). Shared `services/account_management.py` owns the logic; `scripts/reassign_user_id.py` now imports `USER_SCOPED_MODELS` from the service so the cascade table list has one source of truth — caught a missing-table bug while at it (`topic_subscriptions`, added in Phase D, was never in the original 9-table reassign list; now correctly cascaded everywhere). New `/settings/account` page; admin Users tab gains a "Reset password" modal with copy-to-clipboard.

### Beta hardening — rate limit + CSRF + password strength (PR #41)

Three defense-in-depth guards. **Rate limiter** (`backend/middleware/rate_limit.py`) — custom in-memory fixed-window middleware, NOT `slowapi`. Default policies: `POST /auth/login` 5/min/IP, `POST /auth/signup` 3/min/IP, `POST /onboarding/generate-topic` 5/hour/user. Env flag `RATE_LIMIT_DISABLED=1` skips. **Started with `slowapi` per the original plan but its decorator broke FastAPI's pydantic body-parameter introspection** (every POST with a body model 422'd with `loc=["query","body"]`) — rewrote as middleware, dropped the `slowapi` dep entirely. **CSRF** (`backend/middleware/csrf.py`) — double-submit-cookie pattern, non-HttpOnly `ds_csrf` cookie set on every response that lacks one, `X-CSRF-Token` header required on POST/PUT/PATCH/DELETE. Env flag `CSRF_DISABLED=1` skips. Frontend `fetchAPI` auto-attaches the header on mutating requests, with one-shot retry on the warmup case (first request before the cookie is set). **Password strength UI hint** — `<PasswordStrength>` component with length-tier + character-class scorer, no `zxcvbn` dep (would've added ~400KB). Drop-ins on signup, change-password, and admin reset forms. **Mount order in `main.py` is intentional**: requests flow `RateLimit → CSRF → routes` (rate-limit cheaply rejects bursts BEFORE the more expensive CSRF check). Both env flags default to `1` in conftest so the test suite doesn't need to dance around them.

### Per-page guided product tours w/ server-side versioned state (PR #42)

Three independent tours — Dashboard (4 steps), Scope (2 steps), Topics (3 steps) — using **`driver.js`** for the spotlight + popover. Migration `0011_tour_state` adds `users.tour_state JSON NOT NULL DEFAULT '{}'` holding `{tour_id: highest_version_seen}`. Adding a fourth tour later is a new key in the JSON map; **no migration needed**. Each tour component is ~40 lines: STEPS + TOUR_ID + TOUR_VERSION constants + one `useDriverTour()` call. **Originally pivoted from `react-joyride` 2.x** which imports React-18-only APIs (`unmountComponentAtNode`, `unstable_renderSubtreeIntoContainer`) that React 19 (Next 16) dropped — webpack build failed, no stable react-joyride 3.x; switched to `driver.js` (~5KB vs ~80KB, imperative API, React-19-compatible). `PUT /auth/tour-completed {tour_id, version}` uses `max(current, version)` per-key so a stale callback from an older tab can't regress a newer one. `PUT /auth/tour-reset` clears every key — powers the "Show all tutorials again" button on `/settings/account`. `KNOWN_TOUR_IDS = {"dashboard", "scope", "topics"}` server-side; unknown ids rejected with 400 so typos can't pollute the JSON. **`localStorage` removed entirely** — single source of truth on the server, cross-device sync, survives browser-data clears. Bumping a tour's `TOUR_VERSION` re-triggers it for everyone whose stored value is lower; bump only when STEPS materially changes (added/removed step, rewritten copy — not cosmetic tweaks).

## Migration shape

**Eight new migrations.** Apply in order via `alembic upgrade head` (the app's `create_tables()` does this automatically on startup):

1. `0004_notification_settings.py` — `user_settings.notification_settings JSON DEFAULT '{}'`
2. `0005_users_and_sessions.py` — full `users` table (email + user_id + password_hash + status + role + onboarded + timestamps) + `sessions` table (token + user_id FK + expires_at + revoked_at + UA/IP)
3. `0006_invite_codes.py` — `invite_codes` table (code + created_by + expires_at + max_uses + uses + revoked_at)
4. `0007_topic_ownership.py` — `topics.owner_user_id INTEGER NULLABLE` (FK to `users.id` on Postgres, plain int on SQLite) + `topics.visibility VARCHAR(20)` (backfill `'public'`, default `'private'` for new rows)
5. `0008_topic_subscriptions.py` — `topic_subscriptions` table (user_id string + topic_id FK + UNIQUE(user_id, topic_id))
6. `0009_user_onboarded.py` — `users.onboarded BOOLEAN` (server_default `true` so existing rows backfill to onboarded; new INSERTs default `false` via the model)
7. `0010_admin_audit_log.py` — `admin_audit_log` append-only table with denormalized actor + target strings (FK to `users.id` SET NULL on delete)
8. `0011_tour_version_seen.py` — `users.tour_state JSON DEFAULT '{}'` (filename is misleading — see "Worth knowing" below)

**Backfill behavior worth knowing:**
- Existing topics → `owner_user_id NULL` + `visibility 'public'` so Grace's praxis topics stay visible to every account.
- Existing users → `onboarded true` (admins don't get bounced through the wizard) but `tour_state '{}'` (they'll see each new tour once on next visit to its page).
- All 9 pre-existing user-scoped tables keep their `user_id String(100)` columns untouched — the new `users.user_id` matches the same string they already store.

**Filename gotcha (cosmetic, not functional):** `0011_tour_version_seen.py` is the file that actually adds the JSON `tour_state` column — original draft used an INT column called `tour_version_seen` and was rewritten in place before the migration was ever applied anywhere. Sibling `0011_tour_state.py` exists as a no-op placeholder revision (`0011a_placeholder`) so alembic doesn't crash on the orphaned `.py` file. Safe to clean both up on the host filesystem in a follow-up commit if you want — alembic walks the chain by revision id, not by filename.

## Configuration

**New env vars (production should NOT set these — defaults are correct for prod):**
- `OPEN_SIGNUP` — set to `1` only in local dev / CI conftest if you want signup to skip the invite-code requirement. Production stays gated.
- `RATE_LIMIT_DISABLED` — set to `1` in local dev / CI to bypass the rate limiter. Default = enabled (i.e. limits enforced).
- `CSRF_DISABLED` — set to `1` in local dev / CI to bypass CSRF validation. Default = enabled.
- `SESSION_COOKIE_SECURE` — optional override; defaults to `not debug` so prod auto-enables Secure.

**New pip deps** (`requirements.txt`):
- `passlib[bcrypt]>=1.7.4,<2`
- `bcrypt>=4.0,<5`

**New npm dep** (`frontend/package.json`):
- `driver.js@^1.6.0` (slowapi was added and removed within the same release — net-new is just driver.js)

**Bootstrap step after first deploy:** `python scripts/create_admin.py --email <you> --password <pw>` to seed the first admin. From inside the backend container if running compose: `docker compose exec backend python scripts/create_admin.py ...`.

**No new GitHub Actions secrets.** The `RAILWAY_TOKEN_{DEV,PROD}` + `RAILWAY_{BACKEND,FRONTEND}_SERVICE_ID_{DEV,PROD}` matrix is unchanged from v2.0.

## Breaking-ish behavior changes worth flagging

- **`/admin/*` now requires admin role.** Before: any CF-Access-authenticated user could call admin endpoints. After: caller must have a `role='admin'` User row. Solo `__local__` is still treated as admin. **Action**: seed your own admin via the bootstrap script before pointing the prod frontend at the new backend, or you'll lock yourself out of the admin UI.
- **Topic discovery rule changed.** Before: every public topic from any user auto-appeared in everyone's scope + topic catalog. After: users see system topics + their own + topics they explicitly subscribed to via `/topics/discover`. **Action**: existing beta users whose scope relied on auto-seeing other users' public topics need to subscribe via Discover after this ships. Your own topics + system topics are unaffected.
- **`/auth/signup` requires `invite_code`** unless `OPEN_SIGNUP=1`. Production deploys should NOT set the flag.
- **Tour replay on first /-visit after deploy.** Existing users get `tour_state = '{}'` from migration 0011's backfill, so they'll see each tour once on next visit to its page. Suppress with `UPDATE users SET tour_state = '{"dashboard": 1, "scope": 1, "topics": 1}';` post-migration if undesirable.
