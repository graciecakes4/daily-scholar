# 2026-06-25

Setup-audit release. v1.1 got the app onto Railway + Cloudflare; v2.0 makes the prod deploy actually survive the operator misconfigurations that were waiting to bite. **Multi-origin CORS** (no more single-`FRONTEND_URL` trap), **build-time hard-fail** when `NEXT_PUBLIC_API_URL` is missing (so the frontend can never silently bake `http://localhost:8000` into a prod bundle again), **LLM-failure cache-poisoning defense** (a swallowed Gemini 503 used to lock the Topic Review tab on an empty card until the next daily regen — now it surfaces the exception class + traceback and refuses to cache the empty payload), and a **CI workflow fix** (the frontend deploy job was `cd`'ing into `frontend/` before `railway up`, conflicting with the Railway service's `Root Directory=/frontend` dashboard setting). Plus a **README split** that pulls the 818-line monolith into four focused docs (`API.md`, `ARCHITECTURE.md`, `DEPLOY.md`, `PWA.md`). No migrations. Three PRs (#30, #32, #33) closed; ~860 lines changed across 17 files.

## Highlights

**Multi-origin CORS allowlist**
- New `CORS_ALLOWED_ORIGINS` setting on `backend/config.py` (comma-separated string). When set, REPLACES the single-origin `FRONTEND_URL`-derived allowlist; otherwise the existing `FRONTEND_URL` + localhost variants behavior is preserved.
- New `_resolve_cors_origins()` helper in `backend/main.py` strips surrounding whitespace + trailing slashes, dedupes, drops empties — `credentials: 'include'` requires byte-for-byte origin match and the old single-string trap silently rejected prod requests if `FRONTEND_URL` was off by one character.
- `.env.example` grew a CORS section + a Frontend Build-Time Config section documenting both the prod-mode value (`https://scholar.daily-scholar.com`) and the local-dev fallback (`FRONTEND_URL=http://localhost:3000`).

**Build-time NEXT_PUBLIC_API_URL hard-fail (frontend)**
- `frontend/Dockerfile` now `exit 1`s the build when `NEXT_PUBLIC_API_URL` is empty at `next build` time, with a clear error pointing at the Railway service-vars step. The silent fallback to `http://localhost:8000` in `lib/api.ts:7` (which baked a broken URL into the client bundle when the build arg wasn't propagated) can't recur — a misconfigured deploy errors at image-build time instead of shipping a frontend that fetches the user's own machine.
- `frontend/railway.toml` documents the required service variable at the top of the file with the exact prod value.
- `docker-compose.yml` moves `NEXT_PUBLIC_API_URL` from runtime `environment:` to `build.args:` so `docker compose up --build` after this release still works under the new hard-fail check.

**Frontend localhost hardcodes removed**
- `app/page.tsx` quiz-submit switched from a hardcoded `http://localhost:8000` `fetch` (with no `credentials: 'include'`) to the shared `submitAnswer()` helper in `lib/api.ts` — same call path as every other API request, picks up the 401 boundary + the credentials cookie for free.
- `app/layout.tsx` API Docs nav link uses `${API_BASE}/docs` instead of the hardcoded localhost.
- `lib/api.ts` now exports `API_BASE` so layout + any future server-component can reuse the same fallback without re-implementing it.

**LLM cache-poisoning defense**
- `content_generator.py` LLM-calling methods (`generate_paper_summary`, `generate_topic_review`, `generate_quiz_questions`) now log with `type(e).__name__: {e}` + full traceback. Previously the print was `f"Error generating topic review: {e}"` — many exception classes stringify to nothing useful, so silent failures were genuinely invisible in container output.
- `generate_paper_summary` + `generate_topic_review` mark failures with a `__generation_failed__: "<ClassName>: <message>"` sentinel on the returned payload.
- `daily_content.py` checks the sentinel and refuses to cache failed topic reviews. The existing `len(cached_topic_reviews) == 0` retry path then naturally re-runs on the next request instead of locking the user in an empty-card state for 24h until the daily regen.
- Diagnosed root cause for the empty-card incident that triggered the audit: `LLM_TASK_REVIEW=gemini:gemini-3.5-flash` in the env (typo for `gemini-2.5-flash` — Gemini has no 3.5 series). API returned 503 UNAVAILABLE, exception was swallowed, empty payload cached. Fixed env-side (no code); the code change ensures the next env typo surfaces loudly instead of poisoning the cache.

**CI workflow uploads repo root for both services (deploy.yml)**
- The frontend deploy job was `cd`'ing into `frontend/` before `railway up`, so the snapshot uploaded was just the frontend folder. The Railway frontend service has `Root Directory = /frontend` configured in the dashboard — Railway then looked for `frontend/frontend/Dockerfile` inside the snapshot and errored with "could not find the /frontend root directory."
- Backend already used `WORKDIR="."` (upload repo root) and worked fine. Made the frontend symmetric: both services upload from the repo root, Railway's per-service `Root Directory` setting picks the correct subfolder. Inline comment in `deploy.yml` documents why so the next person doesn't reintroduce the `cd`.

**README split into focused docs**
- README went from 818 lines to ~64 — a fork-first quickstart with deep links into the new `docs/` files.
- `docs/API.md` — full endpoint reference (paper discovery, topic CRUD, archive, quiz, daily content, push, user scope).
- `docs/ARCHITECTURE.md` — runtime topology, the topic model, the daily-content cache invalidation flow, the storage abstraction, the auth resolution order.
- `docs/DEPLOY.md` — Railway-specific setup walkthrough (service creation, env-var matrix, Cloudflare Access bundling).
- `docs/PWA.md` — service-worker registration story, push subscription lifecycle, install-prompt rules.

## Configuration

- **No new dependencies.**
- **New env vars (all optional; defaults preserve v1.1 behavior):**
  - `CORS_ALLOWED_ORIGINS` — comma-separated list of allowed origins for the FastAPI CORS middleware. When unset, falls back to `FRONTEND_URL` + localhost variants (v1.1 behavior). Prod recommendation: `CORS_ALLOWED_ORIGINS=https://scholar.daily-scholar.com`.
  - `NEXT_PUBLIC_API_URL` — required at `next build` time for any non-localhost deploy. The frontend Dockerfile hard-fails when empty. Prod recommendation: `NEXT_PUBLIC_API_URL=https://api.daily-scholar.com`.
- **Cloudflare Access topology change required for cross-host cookies.** If the frontend (`scholar.daily-scholar.com`) and backend (`api.daily-scholar.com`) are configured as SEPARATE CF Access applications, the browser session cookie won't ride along on cross-origin API calls and every request gets the Access login redirect (which itself has no CORS headers — manifests as a confusing CORS error in DevTools). Fix dashboard-side: bundle both hostnames under a single CF Access application so they share a session. See `docs/DEPLOY_CLOUDFLARE.md` for the walkthrough.
- **GitHub Actions secret matrix unchanged** — `RAILWAY_TOKEN_{DEV,PROD}` + `RAILWAY_{BACKEND,FRONTEND}_SERVICE_ID_{DEV,PROD}` are the same as v1.1.

## Coming next

The setup audit is closed. Followups captured during the phase:

- **Admin role gate.** `/admin/*` endpoints have no in-app role check; access is gated only by Cloudflare Access. Don't open `/admin/*` to the beta cohort until a role/group gate lands. (Tracked under Phase 4 from the original PWA migration plan.)
- **Local-mode parity test.** The new build-time hard-fail means anyone running `docker build` directly (no `--build-arg`) gets a build failure. `docker compose up --build` passes through the build-arg via the v2.0 compose change, so that path stays smooth — but if a beta tester invokes `docker build` directly they'll hit the hard-fail. Worth a paragraph in `docs/DEPLOY.md` once the first tester trips on it.
- **Cache-poisoning defense for `generate_paper_summary`.** v2.0 wires the sentinel + retry only for topic reviews. The same pattern can extend to paper summaries — caller-side check in `daily_content.py` after `await generator.generate_paper_summary(paper)`, skip the `paper_summary` cache write when the sentinel is present. Lower priority because the cache invalidation key is different (paper is cached when `cached_paper is None`, not on length-zero), so a swallowed paper-summary failure doesn't lock the UI the same way the topic-review one did.
- **`make compose-up` shortcut.** The Makefile only covers local-mode (venv + start.sh). A compose target would help beta testers exercise the Docker path without memorizing `docker compose up --build`.
